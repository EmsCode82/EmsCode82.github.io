Learning notes

DTOs vs Entities (why bother?)

Entities map to database tables and include persistence concerns (keys, relationships). DTOs are slim request/response shapes you expose over the API. DTOs help you: 1) avoid over-posting, 2) version your API without breaking your DB schema, 3) keep validation at the edges. 

public record TodoDto(int Id, string Title, bool IsDone);
          public record TodoCreateDto(string Title);
          public record TodoUpdateDto(string Title, bool IsDone);
EF Core: migrations in plain English

A migration is a tracked snapshot of schema changes. Typical flow:

// Create a migration from current model
          Add-Migration Init

          // Apply to the configured database
          Update-Database

          // If you need to tweak before applying
          Remove-Migration  // (only if not applied yet)

          // Seeding: run code on startup to insert sample rows
CRUD endpoints – predictable shape 

          GET    /api/todos          // list
          GET    /api/todos/{id}     // by id
          POST   /api/todos          // create (TodoCreateDto)
          PUT    /api/todos/{id}     // update (TodoUpdateDto)
          DELETE /api/todos/{id}     // delete

Keep request bodies small (DTOs), validate at the controller boundary, and return 201 Created with CreatedAtAction on POST.

LocalDB vs full SQL Server

LocalDB is a developer-friendly SQL Server instance that runs per-user. Your connection string points to (localdb)\MSSQLLocalDB. Later, swap to full SQL Server/Azure SQL by changing the connection string—your EF Core migrations still apply.
